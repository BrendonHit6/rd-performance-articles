## Слайди 1–3. Вступ і зріз екосистеми

**Мета вступу (2–3 хв):** подати 2026-й як переломний рік, а не чергову доповідь «ще одна ротація фреймворків». Меседж: змінилися *фундаменти* — мова, рантайм і тулчейн, — а не лише бібліотеки поверх них.

Ключові цифри, які варто засвоїти (слайд 3):

- **98,7%** сайтів використовують JavaScript — мова нікуди не зникає; питання лише в тому, *у якій формі* ми її використовуємо.
- **~78%** адопції TypeScript; TS став **найактивнішою мовою на GitHub** за кількістю щомісячних контриб'юторів, обійшовши і Python, і сам JavaScript. Чистий JS дедалі частіше сприймають як legacy-вибір для великих застосунків.
- **84 млн** щотижневих завантажень Vite, **46%** нового коду згенеровано AI, **45%** нових React-проєктів на Server Components.

**Обрамлювальна фраза:** «Три роки тому дебати були React проти Vue. У 2026-му дебати такі: чи потрібен нам взагалі build-крок, чи потрібен нам взагалі Node і чи потрібен нам взагалі ESLint. Це зовсім інший рівень змін.»

## РОЗДІЛ I — Ера TypeScript (слайди 4–10)

### Слайд 5. Node.js 24: нативний TypeScript

Найпрактичніша зміна за минулий рік. Node 24 (поточний LTS) виконує `.ts`-файли за замовчуванням: `node app.ts` просто працює — без tsconfig, без білд-артефактів, без watch-mode тулінгу.

**Таймлайн, який варто озвучити** (показує, що це системний крок, а не трюк):

1. Node 22 — введено прапорець `--experimental-strip-types`
2. Node 22.7 — додано `--experimental-transform-types` (обробляє enum/namespace)
3. Node 23 — strip-types без прапорця
4. Node 22.18 / 23.6 — увімкнено *за замовчуванням*
5. Node 25.2 (лист. 2025) — фічу оголошено **стабільною**
6. Node 24 LTS — базовий дефолт для нових (greenfield) проєктів у 2026

**Як це працює під капотом:** Node використовує модуль `amaro` — тонку обгортку навколо трансформ-рушія SWC. Це *стрипер типів*, а не компілятор.

### Слайд 6. Механіка стрипінгу типів

Головна ментальна модель для аудиторії: **типи трактуються як коментарі.** Node замінює анотації, інтерфейси, аліаси типів і дженерики на *пробіли*, тож номери рядків лишаються ідентичними — саме тому не потрібні source maps, а стектрейси лишаються коректними.

**Критичне застереження (повторити двічі — це непорозуміння №1):** **під час виконання перевірки типів немає.** `add("hello", "world")` за сигнатури `add(a: number, b: number)` виконається і поверне `"helloworld"`. Стрипінг типів розділяє дві речі, які `tsc` завжди об'єднував:

- **Валідація типів** → досі `tsc --noEmit`, запускається в CI / pre-commit / IDE
- **Виконання в рантаймі** → тепер обробляється Node нативно

### Слайд 7. Обмеження — список «підводних каменів»

1. **enum / namespace** генерують рантайм-код → потрібен `--experimental-transform-types` (opt-in, більш інвазивний). Рекомендація команді: мігрувати enum на об'єкти `as const`. Приклад:
   ```ts
   // замість: enum Role { Admin, User }
   const Role = { Admin: 'admin', User: 'user' } as const;
   type Role = typeof Role[keyof typeof Role];
   ```
2. **tsconfig.json ігнорується** в рантаймі — аліаси `paths`, зниження до старіших версій JS, декоратори (TC39 Stage 3, ще не нативні) не працюють.
3. **`import type` обов'язковий.** Тип, імпортований як value-імпорт, не буде видалено → рантайм-помилка. Увімкніть `verbatimModuleSyntax: true`, щоб компілятор це вимагав.
4. **Немає JSX.** Це фіча для бекенду/скриптів/CLI. Фронтенд і далі йде через Vite.
5. **`erasableSyntaxOnly: true`** (TS 5.8+) — прапорець компілятора, що забороняє будь-який синтаксис, несумісний зі стрипінгом. Вмикайте в кожному сервісі, який запускає `.ts` нативно; він перетворює рантайм-сюрпризи на помилки на етапі компіляції.
6. Node також за замовчуванням не запускає TypeScript-файли всередині `node_modules` — публікуйте пакети як `.js`.

### Слайд 8. Цифри cold start

На простому Express-сервері (~600 рядків коду) розрив cold-start між `ts-node` і нативним strip-types становить приблизно **9×** (≈1,4 с → менш ніж 200 мс). Чому: `ts-node` запускає повний TypeScript-компілятор на кожен старт; лоадери `tsx`/SWC пропускають перевірку, але все одно парсять і генерують код; нативний стрипінг живе всередині рантайму, ближче до no-op.

**Переклад мовою бізнесу:** для serverless кожен cold start тарифікується і впливає на користувача. Менш ніж 200 мс проти 1,4 с — це напряму вимірювана різниця у вартості й UX. Додаткові виграші: менше dev-залежностей (менша площа supply-chain атак), простіші Dockerfile (без білд-стадії для внутрішніх інструментів), швидший локальний фідбек.

### Слайд 9. Рекомендоване налаштування

```jsonc
// package.json
{
  "scripts": {
    "dev": "node --watch src/index.ts",   // миттєвий цикл, без компіляції
    "typecheck": "tsc --noEmit",           // ворота коректності
    "build": "tsc"                          // артефакти для прод / публікації
  }
}
```

Практичні правила:

- **Скрипти, CLI-інструменти, внутрішні сервіси, dev-цикл** → запускати `.ts` напряму.
- **Опубліковані npm-пакети** → досі постачати скомпільований `.js`.
- **Продакшн, критичний до затримок** → попередньо скомпільований JS стартує трохи швидше за стрипінг на старті; лишіть білд для гарячого шляху, якщо це важливо.
- Express і Fastify працюють із нативним стрипінгом без змін.

### Слайд 10. TypeScript 7 «Corsa»

Microsoft переносить весь компілятор + мовний сервіс із TypeScript (self-hosted) на **Go**. Цифри з нативних прев'ю: ~10× швидша компіляція, майже миттєві інкрементальні білди, ~8× швидший старт редактора/LSP, ~50% менше пам'яті. Реліз орієнтовно на середину/кінець 2026.

**TS 6.x** (початок 2026) — останній реліз на старій кодовій базі («Strada»), перехідна версія з deprecation-попередженнями. Що робити:

- увімкнути прапорець `--deprecation`; спланувати міграцію від `baseUrl`, `moduleResolution: node`, non-strict режиму, ES5-таргетів;
- вже зараз протестувати `@typescript/native-preview` у CI-джобі, щоб рано зловити проблеми зі стороннім тулінгом.

**Головна теза, яку треба донести:** TypeScript більше не «мова, що компілюється в JS». Він стає *діалектом, який JS-рантайми розуміють нативно*. Пропозиція TC39 **Type Annotations** (Stage 1) дозволила б рушіям ігнорувати синтаксис типів на рівні специфікації — це довгостроковий шлях конвергенції.

## РОЗДІЛ II — ES2026 (слайди 11–15)

Контекст: TC39 схвалив кандидата ES2026 у **березні 2026** (формальне затвердження Ecma GA — у червні). Усе нижче — або Stage 4, або фактично зафіксоване.

### Слайд 12. Temporal

Флагманська фіча — заміна 30-річному `Date`, чиї вади (мутабельність, відсутність часових поясів, заплутана індексація місяців, дивацтва парсингу) змушували всіх переходити на moment/date-fns/luxon.

Що приносить Temporal:

- **Незмінні типи:** `PlainDate`, `PlainTime`, `PlainDateTime`, `ZonedDateTime`, `Instant`, `Duration`. Кожна операція повертає новий об'єкт.
- **Часові пояси й календарі як першокласні сутності**, включно з негрегоріанськими (у парі з новим API `Intl` для ер/monthCode, теж Stage 4).
- **Адекватна арифметика:** `.add()`, `.until()`, `.since()`, `.round()` з явними одиницями.

Демо-сніпети для live coding, якщо дозволяє час:

```js
const meet = Temporal.ZonedDateTime.from('2026-07-10T15:00[Europe/Kyiv]');
meet.add({ weeks: 2 });                        // новий об'єкт, безпечно щодо DST
Temporal.PlainDate.from('2026-01-01')
  .until('2026-07-02').days;                    // 182
Temporal.Now.zonedDateTimeISO('Europe/Kyiv');   // поточний момент, з урахуванням tz
```

**Вплив на команду:** плануйте прибрати date-бібліотеки в новому коді; очікуйте виграш у розмірі бандла. Для старіших таргетів існує поліфіл.

### Слайд 13. Error.isError + Array.fromAsync

- `Error.isError(v)` — надійна нативна перевірка на помилку. `instanceof Error` не спрацьовує між реалмами (iframes, воркери, vm-контексти) і може бути підроблений через `Symbol.toStringTag`. Це той самий `Array.isArray`, але для помилок.
- `Array.fromAsync(src, mapFn?, thisArg?)` — для `for await...of` те саме, що `Array.from` для `for...of`. Збирає асинхронні ітерабельні (стріми, генератори пагінованих API) у масив одним викликом, з опційним асинхронним маппером.

### Слайд 14. import defer

```js
import defer * as analytics from './heavy';
// поки нічого не виконується
analytics.track('click'); // ← перший доступ до властивості = модуль обчислюється
```

- Модуль (і його підграф) завантажується/парситься, але **відкладає обчислення** до першого звернення до неймспейсу.
- Роб Палмер (співголова TC39): найкорисніше у великих кодових базах із глибокими графами модулів, що коштують сотні мілісекунд на старті — «платиш лише за те, що використовуєш».
- На відміну від динамічного `import()`: доступ лишається **синхронним**, без промісної обв'язки, без рефакторингу в async.
- Суміжна робота «module harmony»: **import text** досяг Stage 3 (імпорт текстових файлів як модулів без білд-тулінгу).

### Слайд 15. Map.getOrInsert + Explicit Resource Management

- **Upsert:** `map.getOrInsert(key, default)` і `map.getOrInsertComputed(key, fn)` замінюють шумний танець has/set/get і роблять один пошук у рушії. Класичне застосування: кеші мемоізації, групування.
- **`using` / `await using`:** прив'язка, чий `[Symbol.dispose]()` / `[Symbol.asyncDispose]()` викликається автоматично при виході зі скоупу — *включно з винятками*. Детермінований клінап для з'єднань із БД, файлових дескрипторів, локів, тимчасових ресурсів. Уже постачається в Chrome, Node, Deno, підтримується в TypeScript і Babel. Подавайте це як: «try/finally, який неможливо забути написати».

Гідні згадки, якщо запитають «що ще є в ES2026»:

- `Math.sumPrecise()` — точне сумування багатьох float-ів;
- `Uint8Array.fromBase64 / toBase64 / fromHex / toHex` — нативна конвертація binary↔string;
- ревайвер `JSON.parse` `context.source` + `JSON.rawJSON` — обробка великих чисел без втрат;
- `Iterator.concat` (послідовне з'єднання ітераторів);
- API варіантів `Intl.Locale` (перший день тижня, вихідні, напрямок тексту для локалі);
- стандартизовано аксесор `Error.prototype.stack` (Stage 2.7 → приземляється).

## РОЗДІЛ III — Тулчейн (слайди 16–19)

### Слайд 17. Vite

- **84 млн щотижневих завантажень**; 98% задоволеності проти 26% у Webpack за опитуваннями розробників.
- React, Vue, Svelte і Shopify обрали його дефолтом. Міграція з Webpack для типового проєкту: зазвичай **1–2 дні**; єдиний блокер — непортовані Webpack-специфічні плагіни.
- **Vite 8 + Rolldown:** Rolldown — це Rust-бандлер, що замінює розділення esbuild(dev)/Rollup(prod) — один рушій для обох, усуваючи неузгодженості dev/prod.
- Корпоративний контекст: **Cloudflare придбала VoidZero**, компанію за Vite — довгострокове фінансування забезпечено. (Примітка: команда Rolldown нещодавно призупинила інтеграцію Rust React Compiler через занепокоєння розміром бінарника — знак того, що вони ретельно контролюють скоуп.)

### Слайд 18. Bun

Найгучніша історія року: **Anthropic придбала Bun у грудні 2025**, після чого AI переписав ~960 тис. рядків вихідного коду з Zig на **Rust** за шість днів (їхня заява; сприймайте як заголовок — інженерна реальність складніша).

Цифри:

- 1,52 млн щотижневих npm-завантажень, ~92 тис. зірок на GitHub (травень 2026)
- Встановлення пакетів у монорепо на 847 пакетів, «холодне»: **npm ~32 с проти bun ~1,2 с**
- **16%** JS-розробників запускали Bun у продакшні
- Продакшн-користувачі: Figma, NYT, Intercom, Slack, Cursor, Claude Code, Midjourney, X

**Наше позиціювання (сформулюйте прямо):** готовий до продакшну для малих/середніх сервісів. Низькоризикові точки входу: `bun install` як пакетний менеджер і `bun test` як раннер — обидва drop-in виграші. Як *основний* рантайм: спершу пілот на некритичному сервісі; Node зберігає перевагу в сумісності з екосистемою й операційній зрілості.

### Слайд 19. Biome

- Лінтер + форматер в одному Rust-бінарнику; **~56× швидший за ESLint**; Biome v2 додав **правила з урахуванням типів без виклику TS-компілятора**.
- Міграція здебільшого автоматизована: `biome migrate eslint` / `biome migrate prettier` перекладають наявні конфіги.
- Аргумент понад швидкість: заміна eslint + prettier + десятка плагінів на одну залежність зменшує площу npm supply-chain — пов'язано з розділом про безпеку.
- Чесне застереження: покриття правил не 1:1 з екосистемою плагінів ESLint; нішеві плагіни (напр., дуже фреймворк-специфічні набори правил) можуть бути відсутні. Перевірте список правил перед міграцією репо, що спирається на кастомні плагіни.

## РОЗДІЛ IV — Фреймворки (слайди 20–24)

### Слайд 21. Чотири теми конвергенції

Усі великі фреймворки зійшлися на: **(1)** дрібнозернистій реактивності (сигнали), **(2)** server-first рендерингу, **(3)** оптимізації, керованій компілятором, з TypeScript як базою, **(4)** робочих процесах з AI-асистуванням. Наслідок: вибір фреймворка у 2026 — це про *відповідність команді й компроміси*, а не про сиру спроможність. Критерії корпоративної оцінки змістилися на стабільність, продуктивність у масштабі, доступність (WCAG 2.2 юридично обов'язковий у багатьох регіонах) і сукупну вартість володіння.

### Слайд 22. React 19

- **React Compiler** авто-мемоізує — зрізає непотрібні ре-рендери на ~25–40% і прибирає більшість ручних `useMemo`/`useCallback`/`React.memo`.
- **Server Components:** поліпшення первинного рендеру приблизно 2,4 с → 0,8 с у наведених випадках (~67% швидше); ~45% нових проєктів адаптують RSC.
- Больові точки, які варто чесно визнати: Context API не перетинає серверну межу; дебаг меж server/client реально складний; екосистема Next.js-центрична (ризик lock-in), з TanStack Start і React Router v7 як альтернативами, що дозрівають.

### Слайд 23. Сигнали в різних фреймворках

- **Angular 21:** сигнали + zoneless change detection стабілізовано; бандли ~18% легші без zone.js. Реальне застереження: у великих кодових базах zoneless-міграцію оцінюють як багатоспринтову, а не «перемкнув прапорець». Виграші нерівномірні — найбільші для компонентів із частими оновленнями (форми, живі дані), майже нульові для статичних в'ю.
- **Vue 3.6 Vapor Mode:** компілює компоненти в прямі DOM-операції, без virtual DOM — вражаючі демо швидкості монтування.
- **Svelte 5 Runes** (`$state`, `$derived`): явна реактивність замість магії міток `$:`; ламає сумісність зі Svelte 4, але з офіційним codemod; реальна ціна — зсув ментальної моделі.
- **SolidJS:** еталонна реалізація дрібнозернистої реактивності; добре пасує до Astro islands для контентних сайтів, де домінують INP і розмір бандла.
- **Пропозиція TC39 Signals:** примітиви реактивності можуть увійти в саму мову — стежте, але поки не будуйте на цьому.

### Слайд 24. Кінець важкої гідратації

Три моделі для порівняння:

1. **Класична гідратація:** відправити весь JS, «переграти» застосунок на клієнті, перш ніж щось стане інтерактивним. Роздуті бандли → поганий INP.
2. **Острови / дрібнозерниста гідратація (Astro):** JS отримують лише інтерактивні острови, завантажуються покомпонентно.
3. **Resumability (Qwik):** нуль виконання JS на старті; сторінка серіалізує свій стан у HTML і *відновлюється* — код завантажується в момент взаємодії (напр., обробник кліку завантажується при кліку).

**Прив'язка до бізнесу:** INP (Interaction to Next Paint) — Core Web Vital, релевантний для ранжування. Менше клієнтського JS → кращий INP → вимірюваний вплив на SEO і утримання. Це аргумент, який затверджує бюджет.

## РОЗДІЛ V — AI і безпека (слайди 25–27)

### Слайд 26. AI у робочому процесі

- **46%** нового коду згенеровано AI; Copilot, Claude Code, Vercel v0 — стандартні інструменти для скафолдингу, тестів і рутинного коду. Деякі фреймворки (Remix) переробляють абстракції під «AI-first», щоб генератори видавали надійний результат.
- Балансуюча примітка (використайте — це додає довіри): спільнота вже дискутує, чи не пройшов «vibe coding» пік; якість AI-виводу обмежена ясністю кодової бази й розумінням рев'ювера.
- **Базові правила команди:**
  1. TypeScript всюди — типи є контекстом, що тримає і людей, і AI-моделі чесними; менше вигаданих API.
  2. AI прискорює написання; він не замінює архітектурні рішення чи рев'ю. Автор PR володіє кодом незалежно від того, хто його набрав.
  3. Згенерований код проходить ті самі ворота: `tsc --noEmit`, Biome, тести, людське рев'ю. Ніякої швидкої смуги.

### Слайд 27. Безпека

- **React2Shell (CVE-2025-55182):** RCE у Next.js з оцінкою **CVSS 10.0** — критична вразливість серіалізації, що змусила переосмислити межі довіри у full-stack JS (що перетинає межу server/client і як воно десеріалізується).
- **npm supply chain:** хвилі автоматизованих компрометацій упродовж 2025–2026 — сімейство Shai-Hulud самопоширюваних хробаків, що б'ють по популярних пакетах, шкідливі postinstall-скрипти, тайпсквоти.
- **Контртренд zero-dependency:** фічі ES2026 (Temporal, fromAsync, using, кодеки Uint8Array) нативно замінюють lodash/moment-подібні утилітарні залежності → менші бандли *і* менша поверхня атаки.

**Наш мінімальний рівень (зробіть це зобов'язанням, а не порадою):** аудит lockfile у CI (напр., npm audit signatures / socket-подібне сканування), запінені версії з контрольованими оновленнями, ніякого відкладання security-патчів фреймворків, перевага платформних API над мікрозалежностями, політика мінімальних install-скриптів.

## Очікувані питання (Q&A)

**П: Чи можемо ми повністю видалити build-крок?**
В: Для скриптів, CLI і внутрішніх сервісів — так, уже сьогодні. Для опублікованих пакетів — ні, постачайте `.js`. Для продакшну, критичного до затримок, — заміряйте; попередньо скомпільований вивід усе ще стартує трохи швидше.

**П: Якщо Node запускає TS, навіщо взагалі тримати `tsc`?**
В: Node видаляє типи, але ніколи їх не перевіряє. `tsc --noEmit` лишається воротами коректності. Коли приземлиться Corsa, ця перевірка стане майже миттєвою, тож пропускати її буде ще менше причин.

**П: Чи варто переписувати наявні сервіси на Bun?**
В: Ніякого переписування. Впроваджуйте інкрементально: спершу пакетний менеджер і тест-раннер (drop-in), рантайм — лише через пілот. Придбання покращує фінансування, але операційна зрілість поки на боці Node для критичних шляхів.

**П: Чи варті Server Components болю міграції?**
В: Для наявних SPA — зазвичай не варте переписування. Для нових проєктів із тиском контенту/SEO/INP — так. А от компіляторна частина React 19, навпаки, вигідна всім майже без витрат на міграцію.

**П: Temporal реально доступний чи це розмова про майбутнє?**
В: Stage 4, у специфікації ES2026; постачається в сучасних рушіях, із надійним поліфілом для відстаючих. Уже зараз безпечно стандартизувати для нового коду — звірте caniuse під вашу конкретну матрицю браузерів.

**П: Чи означають 46% AI-згенерованого коду, що нам потрібно менше розробників?**
В: Це зміщує зусилля з набору на специфікування, рев'ю й архітектуру. Наші ворота (типи, лінт, тести, рев'ю) — це саме те, що тримає ті 46% активом, а не пасивом.

**П: Яка одна зміна з усієї доповіді має найвищий ROI?**
В: Biome + нативний TS dev-цикл. Обидва майже безризикові, займають по дню, і прискорюють кожного розробника щодня.
